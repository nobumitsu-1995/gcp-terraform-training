# Day 1: クラウドとGCPの基礎概念

**ゴール**: クラウドの基本モデルとGCPの全体像を掴む。

## このDayの概要

本格的にGCPを触る前に、まずは「クラウドとは何か」「GCPはどういう構造になっているのか」という土台を固めます。具体的なコマンド操作は次のDay以降で扱うため、ここでは用語と全体像を理解することがゴールです。

この項では、次の流れで学びます。

1. **オンプレミスとクラウドの違い** — そもそもクラウドとは何か、サービスモデル（IaaS/PaaS/SaaS等）の違いを押さえる
2. **GCPの基本コンセプト** — プロジェクト・リージョン・課金アカウントといったGCP共通の概念を理解する
3. **リソース階層** — 組織〜リソースまでの階層構造と、権限・課金の境界を把握する
4. **主要サービスの俯瞰** — 本研修で使うサービスがそれぞれ何の役割を持つかを一覧で掴む
5. **IAMの基本** — 誰に何を許可するかを決める認可の仕組みを理解する

最後の「確認課題」で、実際にGCPコンソールを開いてプロジェクト作成までを体験します。

## 学習トピック

### 1. オンプレミスとクラウドの違い

物理サーバーを社内で運用する「オンプレミス」と、Googleなどが運用する設備を借りる「クラウド」では、責任範囲とコスト構造が大きく異なります。クラウドはさらに以下のサービスモデルに分類されます。

| モデル | 例 | ユーザーの責任範囲 |
| --- | --- | --- |
| **IaaS** (Infrastructure as a Service) | GCE, EC2 | OS・ミドルウェア・アプリ全部 |
| **PaaS** (Platform as a Service) | App Engine, Cloud Run | アプリのみ |
| **SaaS** (Software as a Service) | Gmail, Workspace | 利用のみ |
| **CaaS** (Container as a Service) | GKE, Cloud Run | コンテナイメージ |

### 2. GCPの基本コンセプト

- **プロジェクト**: GCPリソースを束ねる単位。課金・IAM・APIの設定がプロジェクト単位で行われる。
- **リージョン / ゾーン**: 地理的なロケーション。リージョン = `asia-northeast1`（東京）、ゾーン = `asia-northeast1-a` のようにリージョン内の物理データセンター。
- **課金アカウント**: 支払い情報を持つアカウント。複数のプロジェクトに紐付け可能。

### 3. リソース階層

```
組織 (Organization)
  └ フォルダ (Folder)        ← 部門・チーム単位での権限管理に使う
      └ プロジェクト (Project)  ← リソースとIAMの境界
          └ リソース (Resource)   ← GCE, GCS, Cloud Run 等
```

個人のGoogleアカウントには「組織」がない場合があります。その場合は「プロジェクト」が最上位の単位となります。

### 4. 主要サービスの俯瞰

本研修で使うサービスを中心に整理します。

| カテゴリ | サービス | 役割 |
| --- | --- | --- |
| Compute | **Cloud Run** | コンテナをサーバーレス実行 |
| Compute | GCE | 仮想マシン（IaaS） |
| Compute | **Cloud Functions** | イベント駆動の小さな関数 |
| Storage | **Cloud Storage (GCS)** | オブジェクトストレージ |
| Storage | **Cloud SQL** | マネージドRDB |
| Storage | **BigQuery** | データウェアハウス |
| Networking | **VPC** | 仮想ネットワーク |
| Networking | **HTTP(S) LB** | ロードバランサー |
| Networking | **Cloud CDN** | CDN |
| Networking | Cloud DNS | DNS |
| Networking | Cloud NAT | プライベートサブネット→外部通信 |
| Messaging | **Pub/Sub** | メッセージング |
| Security | **IAM** | 認可 |
| Security | **Secret Manager** | 秘密情報管理 |
| Security | **Artifact Registry** | コンテナイメージ管理 |

### 5. IAMの基本

**IAM (Identity and Access Management)** とは、ひとことで言うと「**誰が・どのリソースに対して・何をしてよいか**」を管理する仕組み（＝認可の仕組み）です。

#### なぜ必要なのか

クラウドのリソースは、設定次第でインターネット越しに世界中からアクセスできます。もし誰でも何でも操作できてしまうと、

- 知らない人にサーバーやデータを消される、覗かれる
- チームメンバーが誤って本番環境のリソースを壊してしまう
- 乗っ取られたアカウントから大量のリソースを作られ、高額な請求が来る

といった事故や不正に直結します。IAMは「鍵」と「合鍵の権限」を細かく管理するイメージで、こうしたリスクを防ぎます。

#### 何のために・どう使うのか

「**この人（やアプリ）には、この操作だけを許可する**」というルールを設定するために使います。GCPではこのルールを次の3要素の組み合わせ（＝**ポリシー**）で表現します。

- **プリンシパル**: 誰が（ユーザー、グループ、サービスアカウント）
- **ロール**: 何を許可するか（権限のセット。例: 「ストレージの読み取りだけ」）
- **リソース**: どこに対してか（プロジェクト、特定のバケット、など）

> 例: 「`alice@example.com`（プリンシパル）に、`my-bucket`（リソース）への `閲覧`（ロール）を許可する」

実際の設定は、GCPコンソールの「IAM と管理」画面や `gcloud` コマンド、Terraform から行います（具体的な操作はDay 2以降で扱います）。

#### 覚えておきたいポイント

- **サービスアカウント**: 人間ではなく、アプリケーション/サービスのためのID。Cloud Run などのワークロードはこのIDの権限で動きます。
- **最小権限の原則**: 必要最低限のロールだけを付与する。「とりあえずOwner（全権限）」は事故のもとなので避けます。

各要素のより詳しい分類（プリンシパル・ロールの種類など）は [supplementary.md](./supplementary.md) を参照してください。
