# Day 1: クラウドとAWSの基礎概念

**ゴール**: クラウドの基本モデルとAWSの全体像を掴む。

## このDayの概要

本格的にAWSを触る前に、まずは「クラウドとは何か」「AWSはどういう構造になっているのか」という土台を固めます。具体的なコマンド操作は次のDay以降で扱うため、ここでは用語と全体像を理解することがゴールです。

この項では、次の流れで学びます。

1. **オンプレミスとクラウドの違い** — そもそもクラウドとは何か、サービスモデル（IaaS/PaaS/SaaS等）の違いを押さえる
2. **AWSの基本コンセプト** — アカウント・リージョン・AZ・課金といったAWS共通の概念を理解する
3. **リソース階層** — Organization〜リソースまでの階層構造と、権限・課金の境界を把握する
4. **主要サービスの俯瞰** — 本研修で使うサービスがそれぞれ何の役割を持つかを一覧で掴む
5. **IAMの基本** — 誰に何を許可するかを決める認可の仕組みを理解する

最後の「確認課題」で、実際にAWSマネジメントコンソールを開いて操作を体験します。

## 学習トピック

### 1. オンプレミスとクラウドの違い

物理サーバーを社内で運用する「オンプレミス」と、Amazonなどが運用する設備を借りる「クラウド」では、責任範囲とコスト構造が大きく異なります。クラウドはさらに以下のサービスモデルに分類されます。

| モデル | 例 | ユーザーの責任範囲 |
| --- | --- | --- |
| **IaaS** (Infrastructure as a Service) | EC2 | OS・ミドルウェア・アプリ全部 |
| **PaaS** (Platform as a Service) | Elastic Beanstalk, App Runner | アプリのみ |
| **SaaS** (Software as a Service) | WorkMail, Amazon WorkSpaces | 利用のみ |
| **CaaS** (Container as a Service) | ECS, EKS, App Runner | コンテナイメージ |

> ☁️ **責任共有モデル (Shared Responsibility Model)**: AWSでは「クラウド**の**セキュリティ（物理設備・基盤）はAWSが、クラウド**内**のセキュリティ（データ・設定・権限）は利用者が責任を持つ」という考え方が基本です。IaaSほど利用者の責任範囲が広く、SaaSに近づくほど狭くなります。

### 2. AWSの基本コンセプト

- **アカウント**: AWSリソースを束ねる単位。課金・IAM・リソースの境界がアカウント単位になる。GCPの「プロジェクト」に近いが、より重い単位。
- **リージョン / アベイラビリティゾーン (AZ)**: 地理的なロケーション。リージョン = `ap-northeast-1`（東京）、AZ = `ap-northeast-1a` のようにリージョン内の独立したデータセンター群。AZを跨いで冗長化することで高可用性を実現する。
- **ルートユーザー**: アカウント作成時のメールアドレスでログインする最上位ユーザー。強力すぎるため日常操作には使わない。
- **請求 (Billing)**: アカウント単位で課金される。複数アカウントの請求は AWS Organizations で1つにまとめられる（一括請求）。

> 💡 GCPでは1つのアカウントの下に「プロジェクト」を軽量に作って分けますが、AWSでは用途ごとに「アカウント」自体を分けるのが一般的です（例: 本番アカウント / 開発アカウント）。

### 3. リソース階層

```
Organization (組織)
  └ OU (Organizational Unit)   ← 部門・環境単位での一括管理に使う
      └ アカウント (Account)      ← リソース・IAM・課金の境界
          └ リソース (Resource)    ← EC2, S3, Lambda 等（リージョン別）
```

個人で始める場合は「Organization」がなく、単一のアカウントだけで使うこともできます。その場合は「アカウント」が最上位の単位となります。

> ⚠️ GCPの「プロジェクト」はリージョンをまたぐ論理的な箱ですが、AWSのリソースの多くは**リージョンごと**に作成されます。「東京リージョンで作ったS3バケットやEC2は、別リージョンのコンソールには表示されない」点に注意してください（IAMなど一部はグローバル）。

### 4. 主要サービスの俯瞰

本研修で使うサービスを中心に整理します。AWS経験者向けにGCPの対応サービスも併記します。

| カテゴリ | サービス | 役割 | 対応するGCPサービス |
| --- | --- | --- | --- |
| Compute | **App Runner / ECS Fargate** | コンテナをサーバーレス実行 | Cloud Run |
| Compute | EC2 | 仮想マシン（IaaS） | GCE |
| Compute | **Lambda** | イベント駆動の小さな関数 | Cloud Functions |
| Storage | **S3 (Simple Storage Service)** | オブジェクトストレージ | Cloud Storage (GCS) |
| Storage | **RDS** | マネージドRDB | Cloud SQL |
| Storage | **Athena / Redshift** | データ分析・ウェアハウス | BigQuery |
| Networking | **VPC** | 仮想ネットワーク | VPC |
| Networking | **ALB (Application Load Balancer)** | ロードバランサー | HTTP(S) LB |
| Networking | **CloudFront** | CDN | Cloud CDN |
| Networking | Route 53 | DNS | Cloud DNS |
| Networking | NAT Gateway | プライベートサブネット→外部通信 | Cloud NAT |
| Messaging | **SNS / SQS** | メッセージング（Pub/Sub + キュー） | Pub/Sub |
| Security | **IAM** | 認可 | IAM |
| Security | **Secrets Manager** | 秘密情報管理 | Secret Manager |
| Security | **ECR (Elastic Container Registry)** | コンテナイメージ管理 | Artifact Registry |

> 📌 **Pub/Sub と SNS/SQS の違い**: GCPの Pub/Sub は「発行/購読」と「キュー」の両方を1サービスで担いますが、AWSでは **SNS（発行/購読）** と **SQS（キュー）** に分かれており、組み合わせて使うのが一般的です。

### 5. IAMの基本

**IAM (Identity and Access Management)** とは、ひとことで言うと「**誰が・どのリソースに対して・何をしてよいか**」を管理する仕組み（＝認可の仕組み）です。

#### なぜ必要なのか

クラウドのリソースは、設定次第でインターネット越しに世界中からアクセスできます。もし誰でも何でも操作できてしまうと、

- 知らない人にサーバーやデータを消される、覗かれる
- チームメンバーが誤って本番環境のリソースを壊してしまう
- 流出したアクセスキーから大量のリソースを作られ、高額な請求が来る

といった事故や不正に直結します。IAMは「鍵」と「合鍵の権限」を細かく管理するイメージで、こうしたリスクを防ぎます。

#### 何のために・どう使うのか

「**この人（やアプリ）には、この操作だけを許可する**」というルールを設定するために使います。AWSのIAMは次の要素で構成されます。

- **プリンシパル**: 誰が（IAMユーザー、IAMグループ、IAMロール、AWSサービス）
- **ポリシー**: 何を・どのリソースに許可するか（JSON形式で記述。「どのアクションを・どのリソースに・許可/拒否」を定義）
- **ロール**: 一時的に権限を引き受けるための仕組み（アプリや他サービスに権限を渡すときの主役）

AWSのポリシーは、次のようなJSONドキュメントで「アクション」「リソース」「効果（Allow/Deny）」を表現します。

```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject"],
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```

> 例: 「`my-bucket` 内のオブジェクトに対して `s3:GetObject`（読み取り）を許可する」ポリシーを、特定のユーザーやロールにアタッチする。

実際の設定は、AWSマネジメントコンソールの「IAM」画面や `aws` コマンド、Terraform から行います（具体的な操作はDay 2以降で扱います）。

#### 覚えておきたいポイント

- **IAMロール**: 人間ではなく、アプリケーション/AWSサービスに権限を渡すための仕組み。Lambda や ECS タスクは、アタッチされたIAMロールの権限で動きます（GCPの「サービスアカウント」に相当）。
- **アクセスキーよりロールを優先**: 長期的なアクセスキーは流出リスクがあるため、AWS内部のワークロードには**ロール**を使い、キーの埋め込みを避けるのが原則です。
- **最小権限の原則**: 必要最低限の権限だけを付与する。「とりあえず `AdministratorAccess`（全権限）」は事故のもとなので、実務では避けます。
- **ルートユーザーを使わない**: ルートユーザーは最強の権限を持つため、普段はIAMユーザー/ロールで作業します。

各要素のより詳しい分類（ポリシーの種類、ユーザーとロールの使い分けなど）と、AWS⇔GCPの対応表は [supplementary.md](./supplementary.md) を参照してください。
